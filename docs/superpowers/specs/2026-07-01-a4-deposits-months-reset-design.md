# A4: задатки, помесячные данные и полный сброс

Дата: 2026-07-01

## Цель

Довести основной A4-калькулятор до рабочего состояния, в котором:

- партнёрская зарплата считается по шкале `45 / 50 / 55 / 60 / 65 / 70 / 80%`;
- по строке точной сделки можно указать расчётный номер задатка;
- правило 50 000 ₽ и исключение для новостройки с одним агентом применяются прозрачно;
- данные разных месяцев изолированы и восстанавливаются без потерь;
- пользователь может полностью удалить сохранения A4;
- живой `table-ledger` получает активный месяц и новые признаки сделок.

## Границы

- Изменения ограничены `domian-calculator-a4`.
- Браузер, интернет и Git не используются.
- `calculateOffice(state)` продолжает принимать состояние одного активного месяца.
- Полная модель распределения комиссии между несколькими участниками сделки не создаётся. Вводимая сумма — комиссия, приходящаяся конкретному агенту.
- `boosted` остаётся стартовым процентом-floor поверх стандартной шкалы; отдельная пользовательская boosted-шкала не вводится.

## Расчётные правила

### Партнёр

Стандартная шкала:

1. 45%;
2. 50%;
3. 55%;
4. 60%;
5. 65%;
6. 70%;
7-й и последующие — 80%.

В начале каждого месяца автоматический счётчик начинается заново.

### Стажёр

Нормативная стажёрская шкала:

1. 30%;
2. 35%;
3. 40%.

Четвёртый и последующие квалифицированные задатки не продолжаются молча по 40%. Для расчёта применяется переход на соответствующую партнёрскую ступень:

- 4-й — 60%;
- 5-й — 65%;
- 6-й — 70%;
- 7-й и последующие — 80%.

Если у стажёра больше трёх квалифицированных задатков, результат агента получает признак предупреждения. В карточке показываются два текста:

> У стажёра указано больше 3 задатков за месяц. По правилам стажёрская шкала заканчивается на 3-м задатке. Переведите агента в статус партнёра или проверьте условия вручную.

> Расчёт применён как для агента, фактически перешедшего на партнёрскую шкалу после 3-го задатка.

Ввод не блокируется.

### Квалификация сделки

Сделка является квалифицированным задатком, если выполняется одно из условий:

- комиссия конкретного агента `>= 50 000 ₽`;
- установлен признак «Новостройка, один агент».

Обычная сделка меньше 50 000 ₽:

- не является задатком;
- оплачивается по базовой ставке 45%;
- не продвигает автоматический счётчик;
- игнорирует расчётный номер задатка.

Для стажёра сохраняется существующее специальное поведение стажёрской шкалы; изменение порога для стажёров не вводится без отдельного нормативного правила.

### Расчётный номер задатка

Для каждой строки точной сделки добавляется опциональное целое число `>= 1`.

- Пустое значение: ставка определяется автоматическим количеством квалифицированных задатков.
- Заполненное значение: для этой строки выбирается соответствующая ступень шкалы.
- Значение `7` и больше даёт 80%.
- Override влияет только на ставку текущей строки.
- Автоматический счётчик следующих строк увеличивается на одну за каждую фактически введённую квалифицированную сделку и не прыгает к максимальному ручному номеру.
- В fixed-режиме фиксированный процент имеет приоритет; номер задатка сохраняется как данные строки, но не меняет fixed-выплату.
- В boosted-режиме номер выбирает стандартную ступень, после чего применяется существующий `startingRate` как floor.

Результат сделки содержит:

- `sourceIndex`;
- `rate`;
- `payout`;
- `rateSource`: `auto`, `manualDepositOrder` или `baseSmallDeal`;
- `depositOrderApplied`;
- `scaleTierApplied`;
- `isQualifiedDeposit`.

## Модель строк сделки

Для обратной совместимости `dealsInput` остаётся массивом сумм. Рядом добавляются:

- `dealDepositOrders`;
- `dealNewbuildSoloFlags`.

Все массивы нормализуются до длины `dealsInput`. При добавлении и удалении строки они изменяются синхронно.

В расчёте сначала строятся связанные записи:

```js
{
  amount,
  sourceIndex,
  depositOrderOverride,
  isNewbuildSolo
}
```

Только после этого строки с пустой или нулевой суммой исключаются. Это предотвращает приклеивание override к другой сделке.

## UI точных сделок

Каждая строка содержит:

- сумму комиссии агента;
- «Расчётный задаток» с placeholder `авто`;
- checkbox «Новостройка, один агент»;
- применённую ставку;
- выплату;
- статус источника ставки.

Подсказки объясняют:

- вводится комиссия именно этого агента;
- обычная сумма меньше 50 000 ₽ оплачивается по 45% и не двигает шкалу;
- для новостройки с одним агентом любая сумма может быть задатком;
- расчётный номер нужен при другом порядке закрытия или замене развалившегося задатка.

## Помесячное хранение

Рабочий `state` сохраняет форму активного месяца и остаётся совместимым с текущим UI и `calculateOffice(state)`.

Persistence использует контейнер draft V2:

```js
{
  version: 2,
  savedAt,
  selectedMonth,
  scratch,
  months: {
    "2026-01": monthState,
    "2026-02": monthState
  }
}
```

`monthState` содержит `expenses`, `agents`, `ownerSales`, `schemeCheck` и все поля сделок. `scratch` хранит форму без выбранного месяца.

Переключение месяца:

1. сохраняет текущую форму в текущий месяц или `scratch`;
2. загружает существующее состояние выбранного месяца;
3. если месяца нет:
   - при первом выборе переносит непустой `scratch`;
   - иначе создаёт пустой месяц;
4. синхронизирует счётчики id;
5. перерисовывает UI и календарный статус;
6. сохраняет весь контейнер.

Старый `domianA4DraftV1` мигрирует так:

- при непустом `selectedMonth` его state становится записью этого месяца;
- при пустом месяце state становится `scratch`;
- старые `dealsInput` сохраняются;
- новые параллельные массивы получают безопасные пустые значения.

Новый основной ключ — `domianA4DraftV2`. Loader сначала проверяет V2, затем V1.

## Очистка

### Мягкая очистка

Верхняя кнопка называется «Очистить текущую форму». Она очищает только активный месяц или `scratch` и сохраняет обновлённый V2-контейнер.

### Полный сброс

Внизу страницы добавляется отдельный danger-блок и кнопка «Удалить все сохранённые данные».

После подробного confirm пользователь вводит слово `УДАЛИТЬ`. При успехе:

- отменяется ожидающий autosave;
- удаляются `domianA4DraftV1`, `domianA4DraftV2`, `domianA4TableSnapshot` и известные A4 month-prefix keys;
- контейнер в памяти обнуляется;
- `state` становится `createBlankState()`;
- UI перерисовывается;
- `hasUnsavedChanges` остаётся `false`;
- `saveDraft()` после reset не вызывается;
- показывается статус: «Все сохранённые данные удалены. Можно обновить страницу и начать заново.»

`localStorage.clear()` не используется.

## Table ledger

`openTableModePage()` сохраняет snapshot только активного `state`, включая `selectedMonth`, `dealDepositOrders` и `dealNewbuildSoloFlags`.

Snapshot получает новую версию; `table-ledger.js` принимает старые версии и новую. При загрузке:

- активный месяц отображается в состоянии ведомости;
- параллельные массивы превращаются в поля объектов `deal`;
- `buildCalculationAgent()` возвращает их обратно в shared `calculateAgent()`;
- точные строки ведомости показывают и позволяют изменять расчётный номер и признак новостройки;
- пустые старые snapshot получают безопасные defaults.

Legacy `table-mode-parity.test.js` сохраняется как регрессионная проверка старого адаптера, но дополнительно создаётся тест живого `table-ledger.js`.

## Ошибки и восстановление

- Невалидный V2 не уничтожает V1.
- Неизвестный месяц нормализуется в пустую строку.
- Ошибка записи `localStorage` сохраняет текущий механизм статуса ошибки.
- Ошибка удаления отдельного ключа не приводит к `localStorage.clear()`; reset сообщает об ошибке и не заявляет об успешном полном удалении.
- Переключение месяца сначала фиксирует текущие данные в памяти контейнера, поэтому потеря формы из-за смены `<input type="month">` исключается.

## Тестирование

Реализация выполняется RED → GREEN отдельными циклами:

1. партнёрская шкала и пример пользователя;
2. квалификация 50 000 ₽ и новостройки;
3. ручной расчётный номер и связь с исходной строкой;
4. предупреждение и ставки стажёра после третьего задатка;
5. нормализация старых агентов и draft V1;
6. изоляция января/февраля и восстановление активного месяца;
7. soft clear и hard reset с mock `localStorage`;
8. snapshot активного месяца;
9. импорт и расчёт живого `table-ledger.js`.

Финальная проверка:

- `node --check` для всех изменённых JS;
- `a4-calculations.test.js`;
- `calendar-policy.test.js`;
- `table-mode-parity.test.js`;
- новый тест `table-ledger`;
- другие найденные локальные тесты A4.

## Критерии приёмки

- Пятый партнёрский задаток получает 65%, шестой 70%, седьмой и последующие 80%.
- Пользовательский пример заканчивается ставкой 65% для `255404`.
- Ручной номер 2 даёт 50%, номера 7 и 10 дают 80%.
- Обычная сделка 49 999 ₽ не продвигает шкалу; новостройка с одним агентом на 30 000 ₽ продвигает.
- Четвёртый стажёрский задаток получает 60% и сопровождается явным предупреждением.
- Январь и февраль хранят разные формы и восстанавливаются после переключения/перезагрузки.
- Старый V1 draft мигрирует без потери данных.
- Полный reset удаляет сохранения и после обновления не восстанавливает черновик.
- `table-ledger` рассчитывает активный месяц с теми же ставками и признаками сделок.
