# Изолированная логика «Заграница / Путешествие» — дизайн

## Цель и границы

Пересобрать только A4-логику поездки так, чтобы её расчёт, состояние и UI не зависели от имени агента, сделок, статуса стажёр/партнёр, обычного `partnerConfirmed`, квартального результата для стипендии, общей доступности мотиваций или заполнения предыдущих шагов.

В scope входят `domian-calculator-a4/assets/js/calculations.js`, `app.js`, `constants.js` и A4-тесты. `index.html` и CSS меняются только при доказанной необходимости. Simple, extended, table-ledger, archive, сделки, роялти, рефералы, шкалы выплат, fixed/boosted и существующая логика стипендии не меняются.

## Бизнес-правило

Состояние решения собственника хранится одним значением:

- `auto` — применяется обязательное правило;
- `forceInclude` — поездка учитывается как исключение собственника;
- `forceExclude` — заработанная поездка не учитывается для конкретного агента.

В режиме `auto` поездка заработана и учитывается только когда одновременно:

1. `halfYearCommission >= 1_600_000`;
2. `travelQuarterPartnershipConfirmed === true`.

`forceInclude` разрешён при любом невыполненном автоматическом условии. `forceExclude` предназначен для автоматически заработанной поездки. UI не предлагает бессмысленные действия: при невыполненных условиях доступно ручное включение, при выполненных — ручное исключение. Если в сохранённых или импортированных данных встретится неподходящее решение, расчёт остаётся детерминированным: `forceInclude` включает поездку, `forceExclude` исключает её, а UI позволяет вернуться в `auto`.

## Расчётная модель

Чистая функция `calculateTravelMotivation(agent)` читает только:

- `halfYearCommission`;
- `travelQuarterPartnershipConfirmed`;
- `travelDecision`;
- `travelPerTrip` и `travelTripsPerYear` из мотивации с текущими значениями по умолчанию.

Она возвращает как минимум:

```js
{
  resultMet,
  preTripPartnershipConfirmed,
  earned,
  decision,
  counted,
  annualAmount,
  monthlyAmount,
  status,
  message
}
```

`earned` означает выполнение обоих автоматических условий. `counted` означает фактическое включение в расходы после решения собственника. `annualAmount` равен `travelPerTrip * travelTripsPerYear`, только если `counted === true`; иначе он равен нулю. `monthlyAmount` равен `annualAmount / 12` и служит месячной долей поездки.

Приоритет расчёта:

1. `forceExclude` → `counted: false`;
2. `forceInclude` → `counted: true`;
3. `auto` → `counted: earned`.

После нормализации `travelDecision` является единственным источником ручного решения. Если в старом или повреждённом состоянии одновременно присутствуют `travelDecision`, `travelOverride` и `travelEnabled`, валидный `travelDecision` имеет безусловный приоритет.

Функция не вызывает `getMotivationEligibility()` и не читает `agent.name`, сделки, `status`, обычный `partnerConfirmed`, `quarterlyCommission`, `quarterlyDeposits`, `travelEnabled`, `travelOverride` или общие override-поля.

## Интеграция с резервом

`calculateMotivationReserve()` получает поездку только из `calculateTravelMotivation()`. Старое выражение `source.travelEnabled && travelAllowed` удаляется из контура поездки. Другие мотивации продолжают рассчитываться прежним способом.

В месячном режиме вклад поездки в резерв равен `annualAmount / 12`; в режиме полной годовой суммы — `annualAmount`. Ручной режим общего резерва остаётся ручным и не маскируется скрытым автоматическим добавлением; при этом карточка поездки продолжает показывать фактический результат и сумму поездки отдельно. Это сохраняет смысл существующего ручного режима и не меняет несвязанные бизнес-правила.

## Состояние и совместимость сохранений

Новые агенты получают:

```js
travelQuarterPartnershipConfirmed: false,
travelDecision: 'auto'
```

При нормализации старого сохранения используется порядок: валидный `travelDecision` сохраняется; иначе `travelOverride === true` мигрирует в `forceInclude`; иначе выбирается `auto`. Старый `travelEnabled` остаётся только legacy-полем и не превращается в `forceInclude`, поскольку прежняя галочка не доказывает осознанного исключения собственника. `preTripQuarterDeposits` больше не участвует в расчёте поездки; поле можно оставить в старых сохранениях как неиспользуемое для безопасной обратной совместимости.

Неизвестное значение `travelDecision` безопасно нормализуется в `auto`.

## UI и live-update

Полугодовой блок содержит:

- ввод результата за полугодие;
- отдельный выбор «Партнёрство по задаткам в квартале перед поездкой подтверждено?» — Да/Нет;
- автономную карточку поездки;
- одно управляющее решение, а не две независимые галочки.

Управление можно представить контекстными кнопками или segmented control с тремя значениями: «По правилу», «Отправить как исключение», «Не учитывать». При невыполненных условиях акцентируется «Всё равно отправить», при выполненных — «Не учитывать поездку». В ручном состоянии обязательно доступно действие «Вернуть расчёт по правилу».

Состояния карточки:

- зелёное: поездка заработана и учтена автоматически;
- зелёное с пометкой исключения: поездка включена через `forceInclude`;
- красное: автоматические условия не выполнены и решение `auto`;
- warning: заработанная поездка исключена через `forceExclude`.

При ручном режиме общего резерва карточка всё равно показывает отдельную расчётную сумму поездки 200 000 ₽ при текущих константах и её фактический статус, даже когда общий итог резерва задан вручную и не складывается автоматически.

Изменение суммы, подтверждения или решения обновляет только карточку поездки, её показатели и итоговые суммы. Большой DOM агента не заменяется во время ввода. Реализация не использует новые или экспериментальные браузерные API.

## Тексты состояний

- Не выполнен порог: «Поездка не заработана: результат за полугодие меньше 1 600 000 ₽.»
- Нет подтверждения: «Поездка не засчитана: не подтверждено партнёрство по задаткам в квартале перед поездкой.»
- Оба условия выполнены: «Агент заработал поездку.»
- Ручное включение: «Поездка включена вручную как исключение собственника.»
- Ручное исключение: «Агент заработал поездку, но она отключена вручную для этого агента.»

## Проверки

TDD-покрытие включает:

- границы `1_599_999`, `1_600_000`, `2_000_000`;
- все четыре сочетания порога и подтверждения в `auto`;
- `forceInclude` для каждого невыполненного условия;
- `forceExclude` для заработанной поездки;
- независимость от имени, сделок, статуса, обычного `partnerConfirmed` и общей eligibility;
- годовую сумму 200 000 ₽ и месячную долю `200_000 / 12` при текущих константах;
- включение/исключение поездки в общем резерве;
- миграцию старого `travelOverride` и нормализацию неизвестного решения;
- сохранение действующей стипендии при 1 300 000 ₽ и подтверждённом обычном партнёрстве;
- неизменность сделок, роялти, рефералов, fixed/boosted и шкал;
- точечный live-update без замены большого DOM.

Финальная автоматическая проверка: `node --check` затронутых JS и тестов, A4-тесты, parity-тесты и поиск повреждённой кодировки. Браузерная проверка выполняется на доступном Chromium. Для Safari/iOS и Chrome/Android предоставляется точный ручной сценарий; абсолютная гарантия для всех версий браузеров не заявляется без фактического запуска на этих устройствах.

## Критерий готовности

Поездку можно полностью рассчитать на агенте, содержащем только поля travel-контура; итог одинаков локально и после публикации тех же статических файлов на GitHub Pages. Автоматическое правило и оба решения собственника дают однозначные, непротиворечивые состояния и корректно меняют расход.
